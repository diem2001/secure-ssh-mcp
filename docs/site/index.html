01 The Problem

Direct SSH connections via the Bash tool expose sensitive customer data to the public Anthropic-hosted AI. Every file read during an SSH session — credentials, config files, logs — passes through the main Claude context and potentially reaches Anthropic's infrastructure.

Risk: A single cat wp-config.php during an SSH session sends database credentials, API keys, and salts to a third-party AI provider. This is a DSGVO and confidentiality violation.

The solution: route all SSH operations through a self-hosted MCP server that contains its own AI agent. Sensitive data never leaves the self-hosted infrastructure. The main Claude instance only ever sees anonymized context.

02 Architecture

Four isolated layers ensure that raw secrets never reach the public AI. Each layer has strict boundaries enforced at the network level.

05 Dual-Layer Anonymization

Every piece of data passes through two independent anonymization layers before leaving the self-hosted infrastructure. Applied before escalation to main Claude and before writing to logs.

Layer 1 — Rule-Based
⚡ Deterministic & Fast
Regex patterns for known sensitive formats: IP addresses, emails, credentials, API keys, connection strings, private key material. Known file patterns: .env, wp-config.php, .htaccess. Replacements: [IP_REDACTED], [EMAIL_REDACTED], ***REDACTED***.
Layer 2 — MiniMax 2.5
🧠 Contextual & Thorough
Second pass over rule-based output. Identifies contextually sensitive content that rules would miss: credentials embedded in shell scripts, tokens in unexpected formats, personal data in log files. Catches the edge cases.

Canary Token Validation

Known fake sensitive patterns are continuously fed through the anonymization pipeline. If a canary token appears unredacted in output, the system triggers an immediate pipeline failure alert. This provides continuous confidence without manual auditing.

🎯
Inject Canary Tokens
Known fake credentials, IPs, and API keys embedded in test payloads at regular intervals.
🔄
Run Through Pipeline
Canaries pass through both anonymization layers under production conditions.
🚨
Validate Output
If any canary token survives anonymization → immediate alert, pipeline halted, session quarantined.

06 Prompt Injection Defense

Remote file content is inherently untrusted. A malicious actor could embed LLM instructions inside a config file or log to manipulate MiniMax 2.5.

  • Strict Context Separation

    System prompt explicitly separates instruction context from data context. The model knows what is a command and what is untrusted input.

  • Untrusted Data Wrapping

    All remote file content is wrapped and labeled as "untrusted external data" before being processed. No raw file content enters the instruction stream.

  • Output Validation Layer

    Checks whether model responses contain content lifted verbatim from files rather than reasoned conclusions. Detects and blocks exfiltration attempts.

  • Constrained Operation Mode

    MiniMax 2.5 operates in data-analysis-only mode. No instruction interpretation from file content. The model reasons about data — it does not execute embedded commands.

07 Logging Architecture

Every SSH session is logged for optimization and auditing — but raw logs are never retained. The pipeline processes, embeds, and purges.

📝
Ingest
Full session input/output captured. All SSH commands and results logged. Everything passes through the dual-layer anonymization pipeline first.
PostgreSQL + JSONB
🔮
Process & Embed
Anonymized session data is converted to vector embeddings for semantic search over past sessions. Enables pattern recognition and optimization.
Qdrant Vector DB
🗑️
Purge Raw Logs
Raw logs are deleted from PostgreSQL after processing. Only anonymized embeddings are retained. Storage limitation enforced by design.
DSGVO Art. 5
Access Control: Logs are restricted to authorized engineers only. Purpose: optimize MiniMax 2.5 tooling and improve the anonymization pipeline. Data never leaves self-hosted infrastructure.

  • Immutable container images

    MCP server can reach target machines. Target machines reject all other SSH sources. MiniMax 2.5 has no external network access.

    MiniMax 2.5 scales separately from the MCP server. During high load, inference capacity increases without affecting SSH orchestration.
